# App registry — protocol adapter plugin model

LSG hosts protocol adapters as **independent, self-contained sub-projects** that lsg-app
manages as plugins. Each adapter ships its own runtime (Python, Node, …), systemd unit,
and Nginx server-block snippet. lsg-app's job is to install, register, surface, and tear
them down — not to know how they work internally.

The contract between lsg-app and an adapter is exactly one file: **`app_manifest.json`**.

## Two layers of "what's installable"

There are two sources lsg-app consults when listing protocols:

1. **`config/protocols-config.json`** — a curated list of *known* protocols with
   pre-defined GitHub repos. These appear in the UI even when not installed.

   ```json
   {
     "opcua":         { "repo": "https://github.com/Faclon-IoT-Team/protocol-opcua-package", "appName": "ioconnect-opcua",     "description": "OPC UA protocol client for data collection" },
     "node-red":      { "repo": "https://github.com/Faclon-IoT-Team/app-nodered-package",    "appName": "ioconnect-nodered",   "description": "Node-Red app for minor calculations" },
     "opc-newpackage":{ "repo": "https://github.com/Faclon-IoT-Team/ioconnect-opcua-python", "appName": "ioconnect-opcua-newpackage", "description": "Testing new opcua packaging format" }
   }
   ```

2. **`config/app-registry.json`** — runtime registry of *currently installed* apps,
   keyed by `appName`, with the manifest data each adapter declared at install time.

The registry is also where the user can install **arbitrary** apps from any GitHub repo —
not just the curated list. The flow is the same; only the lookup differs.

## The manifest contract

Every adapter must produce an `app_manifest.json` at the root of its install directory.
The manifest is **generated by the adapter's `scripts/install.sh`**, not shipped in the
git repo, because some fields (port, env-derived paths) are computed at install time.

Required fields (validated in [appRegistry._validateManifest](../src/services/appRegistry.js#L114)):

```json
{
  "appName":     "opcua-client",
  "displayName": "OPC UA Client",
  "version":     "1.2.0",
  "port":        5000
}
```

Optional fields:

| Field                  | Default       | Meaning |
|------------------------|---------------|---------|
| `description`          | `""`          | Human-readable description for the UI. |
| `uiEnabled`            | `false`       | Whether the adapter exposes a web UI. If `true`, `uiPath` is required. |
| `uiPath`               | —             | Nginx-mounted prefix for the UI (e.g. `/apps/lsg-opcua/`). |
| `apiPath`              | `null`        | Nginx-mounted prefix for the API (e.g. `/apps/api/lsg-opcua/`). |
| `healthCheckPath`      | `/health`     | HTTP path on `localhost:<port>` to probe for `/api/registry/:name/health`. |
| `runtime`              | `null`        | Free-form (e.g. `"python3.11"`, `"node20"`, `"docker"`). Surfaced in the UI. |
| `scripts`              | `{}`          | Map of action → `{ path, requiresSudo? }`. See "Lifecycle scripts" below. |
| `startupDelaySeconds`  | `3`           | How long the install flow waits before deciding the app is up. |
| `requiresSudo`         | `false`       | Whether the app's lifecycle scripts need root. |
| `repo`                 | `null`        | Source repo URL — surfaced in the UI for "this app came from where". |

Example ([apps/ioconnect-opcua/app_manifest.json](../apps/ioconnect-opcua/app_manifest.json)):

```json
{
  "appName": "opcua-client",
  "displayName": "OPC UA Client",
  "version": "1.2.0",
  "description": "OPC UA protocol client for industrial data collection",
  "port": 5000,
  "uiEnabled": true,
  "uiPath": "/apps/lsg-opcua/",
  "apiPath": "/apps/api/lsg-opcua/",
  "healthCheckPath": "/health",
  "scripts": {
    "start":     { "path": "scripts/start.sh",     "requiresSudo": true },
    "stop":      { "path": "scripts/stop.sh",      "requiresSudo": true },
    "restart":   { "path": "scripts/restart.sh",   "requiresSudo": true },
    "status":    { "path": "scripts/status.sh",    "requiresSudo": true },
    "uninstall": { "path": "scripts/uninstall.sh", "requiresSudo": true }
  },
  "startupDelaySeconds": 5
}
```

## Repository conventions

By convention each adapter repo looks like:

```
<repo>/
  app_manifest.json         # generated by install.sh, NOT committed
  scripts/
    install.sh              # required — orchestrates full deployment
    uninstall.sh            # required — reverses install (and Nginx cleanup)
    start.sh, stop.sh, restart.sh, status.sh, [uptime.sh, logs.sh]
  nginx/
    <appName>.conf          # generated; deployed to NGINX_SNIPPET_DIR
  services/
    <appName>.service       # generated systemd unit (gitignored)
  env/
    *.env                   # generated env files (gitignored)
  protocol/   or  src/  ...  # the actual application code
```

`install.sh` is responsible for *everything*:
- creating the venv / installing npm deps / pulling Docker images
- generating the systemd unit and `systemctl enable --now`-ing it
- generating the Nginx snippet and dropping it in `${NGINX_SNIPPET_DIR}`
- emitting `app_manifest.json` at the repo root

`NGINX_SNIPPET_DIR` and `LSG_APPS_HOME` are exported globally to `/etc/environment` by
the lsg-app installer ([scripts/install.sh](../scripts/install.sh)) so adapter scripts
can find them without hard-coding.

## Install flow

[POST /api/polling/protocols/install](../src/routes/protocol.routes.js) → [protocol.controller.js performInstallation()](../src/controllers/protocol.controller.js):

```
1. Resolve identifier (key from protocols-config.json OR ad-hoc {githubRepo, appName})
2. Reject if already in registry
3. Backup any pre-existing config.csv / sys_parameters.json (preserve user data
   across re-installs)
4. Wipe apps/<appName>/ contents
5. git clone https://<token>@github.com/<repo>.git apps/<appName>
6. Validate scripts/install.sh and scripts/uninstall.sh are present
7. Restore backed-up config files
8. Normalize line endings, chmod +x scripts/*.sh
9. exec sudo bash scripts/install.sh — stream stdout/stderr into installationStatus[key].logs
10. Read apps/<appName>/app_manifest.json (FAIL HARD if missing or invalid)
11. appRegistry.register(appName, manifest, installPath) → persists to app-registry.json
12. POST install-callback to ioadmin if toolId+adminUrl provided
```

The endpoint returns `202` immediately with a `statusEndpoint` URL; the UI polls
`/api/polling/protocols/:protocol/status` for live `step` + `logs[]` until `status` is
`completed` or `failed`. Install timeout is 5 minutes — the controller `kill`s the
child process if it runs longer.

On failure: `apps/<appName>/` is `rm -rf`'d, then any backed-up configs are written
back so the user doesn't lose their `config.csv`.

## Uninstall flow

[DELETE /api/polling/protocols/:protocol](../src/routes/protocol.routes.js) →
`uninstallApp(appName, appPath)`:

```
1. Normalize line endings on scripts/*.sh
2. exec sudo bash scripts/uninstall.sh — stream output to uninstallStatus
3. appRegistry.deregister(appName) → removes from app-registry.json
4. rm -rf apps/<appName>/
```

The adapter's own `uninstall.sh` is responsible for:
- stopping and disabling its systemd unit
- removing its Nginx snippet from `NGINX_SNIPPET_DIR`
- removing any /var or /etc files it created

> **lsg-app does not manage Nginx.** Each adapter's install/uninstall is the only place
> that writes or removes Nginx configs. This avoids tight coupling between lsg-app and
> every adapter's URL routing.

## Lifecycle script resolution

Routes like `/start`, `/stop`, `/restart`, `/status`, `/logs`, `/uptime` resolve the
script path via [resolveScriptPath()](../src/controllers/protocol.controller.js#L125):

1. **Manifest-declared** — if the registry entry has `scripts.<action>`, use that
   path (joined onto `installPath`) and honour the `requiresSudo` flag.
2. **Fallback convention** — `<installPath>/scripts/<action>.sh`.

This means an adapter that follows the standard layout works with no manifest
`scripts` entries at all; the manifest is only needed when scripts live outside
`scripts/` or when sudo escalation is required.

## Health probing

`GET /api/registry/:appName/health` does an HTTP `GET` against
`http://127.0.0.1:<port><healthCheckPath>` (default `/health`). It does **not**
distinguish between "service stopped" and "service up but health endpoint broken" —
use `GET /api/polling/protocols/:protocol/status` (which calls the manifest's
`status.sh`) for systemd-level liveness.

## Re-onboarding wipes apps

`POST /api/system/onboard/reset` (and re-running `POST /api/system/onboard` when
already onboarded) iterates `apps/` and uninstalls every directory. See
[onboarding.md](onboarding.md) for the rationale.

## Adding a new protocol to the curated list

Edit [config/protocols-config.json](../config/protocols-config.json):

```json
"my-proto": {
  "repo":        "https://github.com/Faclon-IoT-Team/my-protocol-package",
  "appName":     "ioconnect-my-proto",
  "description": "My new protocol adapter"
}
```

Then `git commit` and ship. The UI will show it as "available" until a user installs it.
No backend change is required — the install flow uses the repo URL directly, the GitHub
token from secrets, and validates the result via the manifest.

## Failure modes & debugging

| Symptom                                          | Cause / where to look |
|--------------------------------------------------|------------------------|
| Install fails with "did not generate app_manifest.json" | The adapter's `install.sh` finished without writing the manifest. Check install logs in `installationStatus[key].logs` returned from the status endpoint. |
| Install fails with "manifest is missing required field: port" | Manifest validation in [appRegistry._validateManifest](../src/services/appRegistry.js#L114). Adapter author must add the field. |
| `apps/<name>/` exists but UI says "not installed" | `app-registry.json` is missing the entry. The registry only loads at boot — re-running install will rewrite it. |
| Adapter UI returns 404 in browser                | Nginx snippet missing or pointing at the wrong port. Check `NGINX_SNIPPET_DIR` for the adapter's `.conf` and reload Nginx. lsg-app cannot fix this — it's owned by the adapter's install script. |
| `git clone` fails with auth error                | `GITHUB_TOKEN` is missing or revoked. Setup wizard collects this; rotate via re-running the wizard. |
| Re-install lost `config.csv`                     | Should be auto-restored. If the failure path triggered, check the cleanup branch of `performInstallation` — backups are restored only when the install dir is recreated after `rm -rf`. |
