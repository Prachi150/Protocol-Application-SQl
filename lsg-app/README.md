# LSG — Linux System Gateway

LSG is an IoT edge gateway that runs on a Linux device. It manages a set of installable
protocol adapter "apps" (OPC UA, Node-RED, …), forwards collected device data downstream
(MQTT / HTTP / Redpanda pipelines), and is remotely managed by a central **ioadmin** platform.

A single device runs:
- a Node.js/Express backend ([src/](src/)) on port `3001`
- a React SPA ([client/](client/)) served by Express in production
- an always-on master MQTT link to ioadmin (commands, heartbeat, onboarding)
- one or more protocol adapter apps under [apps/](apps/), each owning its own port and Nginx config

## Development

### Backend

```bash
npm install
cp config/env.example .env
# Edit .env — see config/README.md for the full variable list. Minimum for local dev:
#   SETUP_COMPLETE=true
#   JWT_SECRET=dev-secret
#   ADMIN_USERNAME=admin
#   ADMIN_PASSWORD_HASH=<bcrypt hash>     # see snippet below
#   GITHUB_TOKEN=<your token>
#   MASTER_MQTT_USERNAME=dev
#   MASTER_MQTT_PASSWORD=dev
npm run dev      # starts on :3001
```

Generate a bcrypt hash for `ADMIN_PASSWORD_HASH`:

```bash
node -e "const b=require('bcryptjs');b.hash('yourpassword',12).then(console.log)"
```

### Frontend

```bash
cd client
npm install
npm start        # CRA dev server on :3000, proxies /api → :3001
```

## Production install

The frontend must be built **before** the installer runs:

```bash
cd client && npm install && npm run build
cd ..
sudo bash scripts/install.sh
```

`install.sh` is interactive by default — it prompts for admin credentials, MQTT passwords,
and a GitHub token, then encrypts them with `age` and starts the systemd service fully
configured. Pass `--defer-setup` to skip the prompts and complete setup in the browser
wizard at `http://<device-ip>` instead.

The installer also **always** prompts for an Nginx **Basic Auth** username + password
(even with `--defer-setup`). These credentials guard the **human surface** of the
gateway — the React SPA, static assets, and protocol-app UIs (`/apps/...`). The
`/api/` location is intentionally exempted with `auth_basic off;` because server-level
Basic Auth otherwise re-prompts the browser on every polling fetch; the API is still
gated by JWT + setupGuard. The htpasswd file lives at `/etc/nginx/lsg-app.htpasswd`
(APR1-MD5, generated by `openssl passwd -apr1`).

The installer:

1. Generates a device-specific `age` key and `JWT_SECRET` / `INTERNAL_API_KEY`
2. Writes safe config to `/etc/lsg-app/config.env`
3. Writes encrypted secrets to `/etc/lsg-app/secrets.env.age`
4. Writes the Nginx Basic Auth htpasswd file to `/etc/nginx/lsg-app.htpasswd`
5. Creates an Nginx site (frontend static + `/api` proxy, server-level `auth_basic`)
   and a systemd unit with an `ExecStartPre` that decrypts secrets to a tmpfs at
   `/run/lsg-app/secrets.env`
6. Adds a sudoers entry so the service can `systemctl restart` itself after the wizard
   completes

See [config/README.md](config/README.md) for the full secrets model and variable reference.

## First-run setup flow

1. `install.sh` writes `SETUP_COMPLETE=false` and a one-time token to `/etc/lsg-app/setup-token`
2. The browser hits `GET /api/setup/status` → React redirects to `/setup`
3. The wizard POSTs credentials to `/api/setup/complete` with the `X-Setup-Token` header
4. [setupService.js](src/services/setupService.js) bcrypt-hashes the password, re-encrypts
   all secrets via `age`, shreds the temp file, and triggers `systemctl restart lsg-app`

While `SETUP_COMPLETE !== 'true'`, [setupGuard.js](src/middleware/setupGuard.js) returns
`503 SETUP_REQUIRED` for every route except `/api/setup/*` and `/api/health`.

## Two MQTT clients — do not confuse them

|                  | [masterMqttClient.js](src/services/masterMqttClient.js) | data forwarder |
|------------------|---------------------------------------------------------|----------------|
| Purpose          | Always-on link to **ioadmin** (commands, heartbeat, onboarding) | Configurable forwarding of **IoT device data** |
| Credentials      | `MASTER_MQTT_*` env vars — set at install, immutable from UI    | Configured via Data Forwarding UI → `config/app-config.json` |
| Topics           | `lsg/<token>/cmd/#`, `…/res/<action>`, `…/heartbeat`            | User-defined |
| Lifecycle        | Starts unconditionally at boot                                  | Active only when forwarding is enabled |

See [docs/architecture.md](docs/architecture.md) for the full design.
